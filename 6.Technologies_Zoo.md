# 🐻🐨🐼 «Зоопарк технологий» (C++, Data Science на Python и веб-сервер на FastAPI ) 

Это стандартная практика в крупном ИТ-бизнесе. Передача данных между языками является главным узким местом (Bottleneck): если сделать её неправильно, система будет сильно тормозить или падать из-за ошибок памяти.

Чтобы преодолеть эти узкие места, команде нужно использовать проверенные индустриальные подходы. 


## Вот 5 лучших способов состыковать технологии без потери скорости.

* * * * *

### ✅Решение 1. Связка C++ и Python через `pybind11` 

Вместо того чтобы компилировать C++ в отдельную программу, ваш C++ инженер компилирует свой код в виде динамической библиотеки (модуля для Python).

-   Как это работает: Библиотека pybind11 позволяет «обернуть» функции C++ так, что для Python они выглядят как родные. Python-разработчик напишет в коде буквально: `import cpp_core`.
-   Как передаются данные: `pybind11` умеет автоматически конвертировать тяжелые массивы данных из C++ (например, `std::vector`) в стандартные массивы Python (NumPy arrays) без лишнего копирования в памяти.
-   Плюс: И C++, и Data Science, и FastAPI работают внутри *одного общего процесса* в оперативной памяти. Это работает мгновенно.

### ✅Решение 2. Обмен данными через общие файлы в RAM (Shared Memory)

Если C++ инженер категорически хочет делать проект в виде отдельной независимой утилиты (исполняемого файла `.exe` / `.bin`), то передавать многочасовые графики ЭКГ через жесткий диск нельзя --- запись и чтение файлов убьют всю скорость.

-   Как это работает: С++ парсит Холтер и сохраняет результат в виртуальный файл прямо в оперативной памяти (в Linux/macOS это делается через `/dev/shm` или `mmap`).
-   Как передаются данные: C++ пишет массив в память, а Data Science скрипт считывает его оттуда.
-   Плюс: Языки разделены, никто ни от кого не зависит, а скорость передачи равна скорости работы оперативной памяти.

### ✅Решение 3. Строгий контракт данных (Zero-Copy Serialization)

Самая частая ошибка «зоопарка технологий» --- когда С++ выдает данные в одном формате (например, структура из байт), а Python ждет другой (например, список чисел). На конвертацию уходит много времени.

-   Решение: BSA (Бизнес-аналитик) должен зафиксировать единый стандарт бинарных данных до начала кодинга. Например, использовать Apache Arrow или Protocol Buffers (Protobuf) от Google.
-   Это технологии, которые позволяют объявить структуру данных один раз, а затем автоматически сгенерировать код для C++ и для Python. Данные передаются «как есть», байт-в-байт, без затрат на парсинг.

* * * * *

Как Проджекту и BSA организовать этот процесс:

1.  Задача для BSA: Прописать форматы на стыках. Например: *«Выходной формат С++ ядра --- массив `float32`, где каждая точка --- это милливольт ЭКГ, с частотой дискретизации 250 Гц»*.
2.  Задача для Проджекта (PM): Выделить первую неделю осени на «Микро-спринт интеграции». Пусть C++ и Python разработчики напишут простейшую функцию (например, C++ просто прибавляет единицу к числу и отдает в Python). Как только этот «мост» заработает, они смогут спокойно писать сложную логику, зная, что зоопарк технологий состыкован.

### ✅Решение 4.  Общие системы сборки и деплоя (Критически важно в РФ)
Пытаться загнать C++ разработчика и Data Science специалиста в одну IDE --- это гарантированный способ усложнить жизнь обоим.

▶️ Идеально для Python и Data Science: Visual Studio Code (VS Code) или его полностью независимый аналог VSCodium. VS Code имеет лучшую в индустрии интеграцию с Jupyter Notebooks (интерактивные графики холтеров, анализ ЭКГ, обкатка моделей) и плагинами для Python.

▶️ Идеально для графической оболочки (GUI) и удобно для С++: (Рекомендуемый для импортозамещения). Медицинский софт требует визуализации: графики ЭКГ, тренды ЧСС, таблицы нарушений ритма. Программы на Qt идеально, «нативно» выглядят в Astra Linux и РЕД ОС, легко проходят сертификацию ФСТЭК и компилируются под любую ОС.  Если графики ЭКГ и окна программы делать на Qt (а для медицинского софта под Linux это стандарт), то дизайн интерфейса (.ui файлы) удобнее всего рисовать во встроенном в Qt Creator дизайнере (Qt Designer). Используйте связку PySide6 / PyQt6. Интерфейс пишется на Python (питонисту это проще), но под капотом работает мощный C++ движок Qt. Вот тут есть нюанс: питонисту и дата-сайентисту работать в Qt Creator будет неудобно. В нем можно открыть Python-файлы, но там нет той глубокой интеграции с виртуальными окружениями (venv/conda) и интерактивными графиками (Jupyter Notebooks), которая жизненно необходима для проверки гипотез Data Science.

Единственное верное решение для «зоопарка» --- это разделение сред разработки при сохранении единой независимой системы сборки.  И этой системой должен стать CMake (файл `CMakeLists.txt`), который понимают и поддерживают абсолютно все современные инструменты. Никаких специфичных для Windows файлов решений (.sln)  или Makefile.

Makefile --- это финальный низкоуровневый сценарий сборки для конкретной ОС, а CMake --- это генератор, который сам создаёт эти файлы под любую ОС и любую IDE.

Если говорить метафорами: Makefile --- это подробная пошаговая инструкция для строителей (компилятора) на конкретном объекте, а CMake --- это архитектор, который чертит план проекта и сам решает, какую инструкцию выдать строителям в России (под Linux), в США (под Mac) или для рабочих с разными инструментами (Visual Studio/ Qt).

#### Вот ключевые отличия по пунктам, которые критичны для стартапа:

1\. Платформозависимость (Самое важное)

-   Makefile: Жестко привязан к платформе. Написанный под Linux файл `Makefile` не запустится на Windows без костылей (нужен MinGW, Cygwin и т.д.), потому что внутри него прописаны конкретные консольные команды терминала Linux (например, `rm -rf`, `mkdir`) и конкретный компилятор (например, `g++`).
-   CMake: Абсолютно кроссплатформенный. Вы пишете один файл `CMakeLists.txt` в универсальном синтаксисе. На Windows он сгенерирует проект для Visual Studio (`.sln`), на Mac --- проект для Xcode, а на Linux --- тот самый `Makefile`.

2\. Поддержка в IDE

-   Makefile: Большинство современных сред разработки (особенно под Windows и Mac) не умеют нормально «из коробки» работать с голыми Makefile. При изменении структуры проекта (добавили новый `.cpp` файл) вам придется вручную переписывать связи в текстовом файле.
-   CMake: Это индустриальный стандарт. Ваш Qt Creator понимает `CMakeLists.txt` как родной. Когда вы нажмете в Qt Creator «Добавить новый класс», среда сама допишет его в CMake. То же самое сделает VS Code у ваших питонистов.

3\. Синтаксис и сложность поддержки

-   Makefile: Требует глубоких знаний работы компилятора и линковщика. Вам нужно вручную прописывать, какие объектные файлы (`.o`) от каких заголовочных (`.h`) зависят. Ошибка в один символ табуляции (вместо пробела) может сломать всю сборку.
-   CMake: Синтаксис более высокоуровневый и декларативный. Вместо ручной настройки флагов компилятора вы просто пишете понятные команды:

    cmake

    ```
          add_library(holter_core SHARED parser.cpp filter.cpp) # Создать библиотеку
          target_link_libraries(holter_core Qt6::Widgets)       # Подключить к ней Qt

    ```

4\. Поиск зависимостей

-   Makefile: Если вашему C++ ядру нужна библиотека (например, для работы с графиками или математикой), в Makefile вам придется вручную прописывать жесткие пути, где она лежит на диске. Если у вас она лежит в `C:/libs`, а на Astra Linux в `/usr/lib`, проект не соберется.
-   CMake: Имеет встроенный мощный механизм поиска `find_package()`. Он сам просканирует систему пользователя (неважно, Mac это или Windows), найдет где установлена библиотека и правильно ее подключит.

#### Резюме:

Если вы выберете Makefile, вам придется писать и поддерживать три разных файла сборки: `Makefile.linux`, `Makefile.mac` и скрипты для Windows.

Выбрав CMake, вы пишете один файл `CMakeLists.txt`. Вы открываете его в Qt Creator и работаете, а ваши коллеги-питонисты используют его же в VS Code для сборки C++ ядра под свои Python-скрипты.


#### Вот как будет выглядеть рабочая схема без конфликтов:

1\. Как распределятся роли и IDE

Каждый член команды работает там, где ему максимально удобно, но проект один:

-   Вы (C++): Работает в Qt Creator (или CLion / VS Code). Пишет высокопроизводительный код для парсинга и фильтрации сигналов ЭКГ.
-   Питонист и Дата-сайентист: Работают в VS Code (или PyCharm). Они пишут логику анализа, подключают нейросети и тестируют алгоритмы в Jupyter-ноутбуках.

2\. Как `CMakeLists.txt` объединяет команду

Вы создаете общую структуру репозитория в Git. Файл `CMakeLists.txt` в корне проекта решает сразу три критические задачи:

   *  Для Qt Creator: «из коробки» считывает `CMakeLists.txt`, понимает структуру проекта, подключает библиотеки Qt и компилирует ваш C++ код в динамическую библиотеку (`.dll` под Windows, `.so` под Linux).
   *  Для питонистов: С помощью расширения pybind11 ваш `CMakeLists.txt` настраивается так, что при сборке проекта C++ код автоматически превращается в готовый Python-модуль (например, `holter_core`).
   *  Автоматизация: Питонисту не нужно лезть в C++. Он нажимает одну кнопку (или пишет `pip install .`), CMake собирает ваш плюсовый движок, и питонист просто пишет в своем коде: `import holter_core`.

3\. Пример правильной структуры проекта

Чтобы всё работало без сбоев на любой ОС (от Windows до Astra Linux), организуйте Git-репозиторий следующим образом:

Если завтра питонист решит уйти на macOS, дата-сайентист останется на Windows, а С++ - на Astra Linux для тестов --- код не сломается. CMake сам определит, какие компиляторы установлены на машине конкретного разработчика, и соберет проект локально.


Docker: Обязательно упакуйте весь готовый Python+C++ стек в Docker-контейнер. Российские больницы часто требуют развертывания софта на их внутренних серверах. Docker-контейнер запустится на любом серверном Linux без боли с зависимостями.

Важный совет по лицензии Qt

Поскольку стартап студенческий и в будущем, скорее всего, возникнет потребность продавать ПО в больницы, используйте бесплатную лицензию Qt Open Source (LGPLv3).Чтобы не нарушать её условия, собирайте ваше C++ ядро и интерфейс в виде динамических библиотек (.dll под Windows и .so под Linux).Не изменяйте исходный код самого фреймворка Qt. В таком случае вы сможете продавать свой продукт, не открывая коммерческий исходный код своей программы.

###  ✅Решение 5.  Кроссплатформенность

<img width="1256" height="739" alt="image" src="https://github.com/user-attachments/assets/c5fa6805-b361-479b-9f68-637fd5a6ae50" />

Общее число пользователей ПК в мире оценивается примерно в 2 миллиарда человек. Статистика использования операционных систем сильно меняется в зависимости от типа устройств.

Рынок персональных компьютеров (ПК и ноутбуки)

Если рассматривать только классические компьютеры, то по данным аналитического агентства *StatCounter*, доли распределяются следующим образом:

-   Windows: ~73% рынка. Абсолютный лидер для работы и игр.
-   macOS: ~14.5% рынка. Операционная система от Apple для компьютеров Mac.
-   Linux (Desktop): ~4% рынка. Популярность этой ОС стабильно растет среди программистов и геймеров.
-   ChromeOS: ~1.5--2% рынка. Система от Google, которая также базируется на ядре Linux.
-   Другие/Неизвестные: ~6% рынка. 

Глобальный рынок (с учетом смартфонов)

Если учитывать абсолютно все умные потребительские устройства (смартфоны, планшеты, компьютеры), картина радикально меняется из-за доминирования мобильных платформ:

-   Linux (включая Android): Свыше 3,5 миллиардов человек. Система Android создана на базе ядра Linux. С учетом этого Linux становится самой массовой ОС на планете.
-   Windows: Те же ~1.4 миллиарда человек. На мобильном рынке Windows практически отсутствует.
-   iOS / macOS (Экосистема Apple): Около 1,5--2 миллиардов человек. Сюда входят все активные пользователи iPhone, iPad и Mac.

Серверный рынок и суперкомпьютеры

В этой сфере Linux занимает доминирующее положение (более 70--80% всех веб-серверов в мире и 100% самых мощных суперкомпьютеров планеты). На серверную версию Windows приходится меньшая часть рынка, а macOS для серверов практически не используется.

В государственных российских больницах и поликлиниках происходит масштабный переход на российские операционные системы на базе Linux. Что касается сложного медицинского оборудования (томографы, аппараты УЗИ, ИВЛ), там обычно встроены закрытые (embedded) версии операционных систем от производителей (часто на базе модифицированного Linux или специализированных ОС). Для планшетов и смартфонов, которые используют выездные врачи или терапевты при обходе, начинает внедряться мобильная российская операционная система «Аврора» (тоже созданная на независимой кодовой базе). Однако в сельских больницах и ФАПах (фельдшерско-акушерских пунктах) Linux используют далеко не все, хотя процесс перехода идет и там. Сельская медицина сейчас находится в состоянии так называемого «лоскутного» ИТ-ландшафта: на одном рабочем месте может стоять современная защищенная ОС, а на соседнем --- старая Windows 7.

Ситуация в регионах и селах зависит от трех ключевых факторов.

1\. Как устроен компьютер сельской больницы

В большинстве сел больницы и амбулатории организационно являются филиалами крупных районных больниц (ЦРБ). 

-   Если ЦРБ централизованно закупила новые компьютеры или обновила парк техники во всех своих подразделениях по госпрограмме, то в сельский филиал привозят моноблоки с уже предустановленным российским Linux (чаще всего Astra Linux или РЕД ОС).
-   Если финансирования не хватило, медики продолжают работать на старых компьютерах с Windows, которые «донашивают» годами.

2\. Главное --- доступ к интернету и МИС

Для сельского врача операционная система не имеет решающего значения. Главное, чтобы на компьютере работал браузер, через который открывается облачная Региональная медицинская информационная система (РМИС).\
Благодаря нацпроектам на селе активно проводят скоростной интернет. Сельский фельдшер должен вносить данные пациента в систему, чтобы они отображались в областной больнице. Браузеру все равно, на чем работать --- на Windows 10 или на Astra Linux. Поэтому в глубинках замена ОС часто откладывается, так как «и на старых окнах база открывается». 
3\. Основные проблемы на селе

Полный переход сельских больниц на Linux тормозит сугубо человеческий фактор:

-   Кадровый голод в ИТ: В селе или маленьком поселке нет штатного системного администратора. Если Linux зависнет, выдаст ошибку или «потеряет» старый принтер, починить его на месте некому. Администратор сидит в ЦРБ за десятки километров и консультирует по телефону или через удаленный доступ, что очень усложняет работу на незнакомой ОС.
-   Цифровая грамотность и возраст персонала: В сельской местности работает много пожилых врачей и медсестер. Переучиваться с привычной Windows (где они знали каждую кнопку) на незнакомый интерфейс Linux для многих становится огромным стрессом. 
Итог: Направление у всех регионов одно --- тотальный Linux. Но если крупные областные центры и городские больницы переведены на него уже на 80--100%, то в сельской местности этот процесс затянется. 

Когда вы пишете кроссплатформенный софт (C++ + Python) для медицинских целей, не зная конечной ОС, главный принцип --- «абстрагироваться от операционной системы». Ваш код должен быть написан так, будто ОС вообще не существует, а все системные вызовы заменены стандартными библиотеками.

### Вот чек-лист критически важных вещей, которые нужно учесть прямо сейчас, на старте архитектуры:

1\. Специфика C++: забудьте про Win32 API и Linux-специфичные функции

-   Пути к файлам: Никогда не пишите пути вручную через слэши (особенно обратные `\` из Windows). Используйте только стандартную библиотеку `std::filesystem` (появилась в C++17). Она сама поймет, где запущен софт, и расставит правильные разделители.
-   Типы данных: Суточная запись Холтера --- это огромный массив бинарных данных. Избегайте базовых типов вроде `long` или `int`, так как их размер (32 или 64 бита) отличается в компиляторах под Windows (MSVC) и Linux/macOS (GCC/Clang). Используйте строго фиксированные типы из `<cstdint>`: `int32_t`, `uint16_t`, `double`.
-   Никаких платформенных библиотек: Забудьте про `windows.h` или `unistd.h`. Если нужен многопоточный парсинг сигналов ЭКГ, используйте кроссплатформенный `std::thread`, `std::mutex` или библиотеку OpenMP.

2\. Специфика Python и Data Science: бинарные зависимости

-   Питонист и дата-сайентист привыкли ставить библиотеки через `pip install`. Но некоторые тяжелые библиотеки (например, TensorFlow, PyTorch или специфические C-расширения для обработки сигналов типа `scipy`) могут требовать разной компиляции под разные ОС (особенно под чипы Apple Silicon M1/M2/M3 на Mac).
-   Решение: Команда должна жестко фиксировать версии библиотек в файле `requirements.txt` или использовать менеджеры зависимостей вроде Poetry / Pixi, которые умеют собирать кроссплатформенные окружения lock-файлами.

3\. Разный порядок байт (Endianness) --- смертельно для Холтеров

Разные процессоры и ОС могут читать бинарные файлы (например, *.dat или *.edf с прибора Холтера) по-разному: слева направо (Big-Endian) или справа налево (Little-Endian).

-   Если вы будете читать бинарный поток данных ЭКГ «в лоб» через `fread` или `ifstream.read()`, на Windows код сработает отлично, а на каком-нибудь специфичном медицинском терминале или Mac значения вольтажа превратятся в кашу.
-   Решение: В  C++ парсере обязательно должна быть явная функция проверки и конвертации порядка байт (byte-swapping).

4\. Проблема кодировок (Боль всей разработки в РФ)

-   Windows исторически использует кодировки вроде `CP1251` или `UTF-16`, а Linux и macOS --- чистый `UTF-8`. Если ФИО пациента или название больницы содержат кириллицу, при переносе текстовых файлов или баз данных между ОС вы получите «кракозябры».
-   Решение: Внутри программы абсолютно все строки (в C++ и Python) должны быть строго в UTF-8.

5\. Как тестировать софт на «чужих» ОС без покупки трех ноутбуков

Вы не можете выпустить медицинский софт, надеясь, что «на линуксе он как-нибудь запустится сам». Тестировать нужно постоянно в процессе разработки.

-   Для разработки: Используйте Docker. Питонист может упаковать приложение в контейнер на базе Linux, и этот контейнер гарантированно запустится и на его Windows (через WSL2), и на Mac, и в целевой больнице.
-   Для автоматизации (CI/CD): Настройте GitHub Actions или GitLab CI. При каждом вашем коммите в Git робот на удаленном сервере будет автоматически пытаться собрать ваш `CMakeLists.txt` под тремя ОС сразу (Ubuntu Linux, Windows, macOS). Если вы случайно применили Windows-функцию, робот сразу пришлет уведомление: *«Сборка под Linux упала»*.
