Для фронтенд-разработчика на Python (который, скорее всего, будет писать интерфейсы на Streamlit, Dash или Flet) инженер ИАС — это главный поставщик «топлива» (данных).
Интерфейс — это просто красивая оболочка. Чтобы она работала, фронтендер должен точно знать, откуда «притягивать» данные для графиков врача и куда «складывать» заметки из дневника пациента.
Вот как выглядит разделение обязанностей и контракт между инженером ИАС и Python-разработчиком интерфейсов

------------------------------
## Разделение обязанностей (Кто за что отвечает)

| Задача | Инженер ИАС  | Python Frontend Разработчик |
|---|---|---|
| Интерфейс Дневника | Проектирует таблицу Patient_Logs в базе данных. Прописывает типы полей: время начала, время конца, тип активности. | Верстает форму для пациента (экран смартфона или веб-страницу), где человек выбирает время и пишет: «Бег» или «Сон». |
| Отправка дневника в БД | Задает правила: в каком формате база готова принять эти записи от фронтенда, чтобы C++ парсер потом смог их прочитать. | Пишет код (кнопка «Сохранить»), который упаковывает введенные пациентом данные и отправляет их в базу по правилам ИАС. |
| Интерфейс Врача (Графики) | Готовит SQL-запросы, которые за доли секунды вытаскивают из базы суточные метрики, пульс и массив найденных аритмий. | Рисует дашборд для врача: выводит графики ЧСС, таблицы со статистикой и подсвечивает красным цветом зоны аномалий. |
| Оптимизация загрузки | Настраивает базу данных так, чтобы при открытии графиков интерфейс фронтендера не зависал намертво от перегрузки. | Пишет логику отображения: делает так, чтобы врачу было удобно скроллить суточную кардиограмму и скачивать отчеты. |

------------------------------
## Как настроить их взаимодействие через Контракты (Договоры)
ИАС прописывает для питониста-фронтендера два четких контракта: на «вход» (Дневник) и на «выход» (Экран врача).


## Контракт 1: Запись Дневника Пациента (Фронтенд ➔ База данных)
Когда пациент отмечает в приложении, что он пошел на пробежку, фронтенд должен записать это в базу.

* Инженер ИАС задает структуру: Фронтенд должен отправить в таблицу Patient_Logs строго определенный набор полей.
* Питонист-фронтендер обязуется передавать данные в таком формате:

{
  "record_id": 98745,
  "start_time": "2026-03-15 14:00:00",
  "end_time": "2026-03-15 14:30:00",
  "activity_type": "running",
  "comment": "Почувствовал легкую одышку"
}

(C++ инженер заберет этот пакет из базы и использует для фильтрации ложных аномалий).

## Контракт 2: Вывод данных для Врача (База данных ➔ Фронтенд)
Когда врач открывает личный кабинет, чтобы посмотреть результаты анализа Холтера, фронтенд запрашивает данные у системы.

* Инженер ИАС обязуется отдавать данные в таком виде: По запросу конкретного record_id база мгновенно возвращает агрегированный JSON.
* Питонист-фронтендер принимает этот пакет и распределяет по экрану:
* Из блока "metrics" (mean_hr, max_hr, sdnn) — строит верхнюю панель с крупными цифрами-индикаторами здоровья.
   * Из блока "anomalies" (список временных меток вроде [{"start": 3600, "end": 3602, "type": "Экстрасистола"}]) — строит интерактивный таймлайн, где врач может кликнуть на аномалию, и интерфейс автоматически перемотает график ЭКГ на эту секунду.

## В чем профит для стартапа?
Питонисту-фронтендеру не нужно думать, как там внутри считалась аритмия или как C++ склеивал файлы. Благодаря ИАС, фронтендер работает с базой данных как с «черным ящиком»: нажал кнопку — улетели данные дневника, открыл страницу — прилетели готовые графики для врача. Интерфейс пишется очень быстро, что критически важно для MVP в акселераторе.

## ТЕХНИЧЕСКОЕ ЗАДАНИЕ (СПЕЦИФИКАЦИЯ)
Наименование экрана: Личный кабинет врача-исследователя (Экран анализа сессии Холтера)
Компонент системы: Frontend (Python)
Ответственный за архитектуру данных: Инженер ИАС


## 1. Назначение экрана
Предоставление врачу-исследователю визуальных аналитических данных по результатам суточного мониторинга ЭКГ пациента, включая общие метрики здоровья и интерактивную ленту найденных аномалий (аритмий) с привязкой к дневнику активности.


## 2. Структура и элементы интерфейса (Wireframe)
Экран должен быть разделен на 3 логических блока:


## Блок А: Карточка пациента и общие метрики (Верхняя панель)

* Элементы:
* ФИО/ID пациента, возраст, пол (данные из таблицы Patients).
   * 3 крупных виджета-индикатора (Метрики из таблицы Ecg_Metrics):
   1. Средний пульс (ЧСС) за сутки (норма: подсвечен зеленым, отклонение — красным).
      2. Макс/Мин ЧСС (с указанием времени, когда зафиксирован пик).
      3. Индекс SDNN/RMSSD (показатели вариабельности ритма для SAS-исследователей).
  
## Блок Б: Интерактивный график ЭКГ (Центральная панель)

* Элементы:
* Основное поле: Линейный график ЭКГ-сигнала (отрисовывается через библиотеку Plotly).
   * Таймлайн-слайдер: Позволяет врачу «скроллить» сутки или выбирать конкретный час исследования.
   * Цветовая разметка на графике:
   * Участки, где модель Data Science нашла аномалии, должны подсвечиваться красным полупрозрачным фоном (данные берем из таблицы Anomalies по таймштампам).
      * Участки, где пациент делал отметки в дневнике (например, «Бег»), должны подсвечиваться желтым фоном (данные из таблицы Patient_Logs).
  
## Блок В: Лента аномалий и фильтры (Боковая/Нижняя панель)

* Элементы:
* Фильтр по типу события: [ ] Все, [ ] Экстрасистолы, [ ] Фибрилляция, [ ] Артефакты шума.
   * Интерактивная таблица (список): Выводит все найденные моделью аномалии в хронологическом порядке.
   * Критическая логика взаимодействия: При клике на любую строку в таблице аномалий график ЭКГ в Блоке Б должен автоматически сфокусироваться (перемотаться) на точное время (timestamp_start) этой аномалии.

## 4. Требования к производительности (Память и Оптимизация)

* Суточная запись ЭКГ содержит миллионы точек. Запрещено пытаться загрузить весь суточный график в браузер одновременно — интерфейс упадет.
* Реализация питониста: Фронтенд должен запрашивать данные порциями (динамическая подгрузка/пагинация). При масштабе «сутки» показывать усредненный пульс, при масштабе «минута» — подгружать детальные точки сигнала (через функцию, которую подготовит C++ или Бэкенд-разработчик).

## Почему это ТЗ полезно?
Питонист, открыв этот документ, сразу увидит: какие библиотеки использовать (Plotly), из каких полей JSON-пакета брать цифры для виджетов, и как связать клик по таблице с перемоткой графика. Споры внутри команды исчезнут, а ИАС покажет эту схему как пример грамотного системного проектирования.


Чтобы выдать фронтендеру весь необходимый JSON-пакет для экрана врача (метрики, аномалии и дневник пациента), ИАС не нужно писать много мелких запросов. В современных СУБД (например, PostgreSQL) инженер ИАС напишет один эффективный SQL-запрос, который соберет данные из четырех разных таблиц и автоматически упакует их в итоговый JSON-формат.
Это демонстрация хард-скиллов в работе с базами данных (использование JSON-функций и группировок).
Вот как будет выглядеть структура этого SQL-запроса для MVP:

SELECT json_build_object(
    'record_id', r.record_id,
   
    -- 1. Собираем блок информации о пациенте
    'patient_info', json_build_object(
        'age', p.age,
        'gender', p.gender
    ),
   
    -- 2. Собираем блок суточных метрик ЭКГ
    'summary_metrics', json_build_object(
        'mean_hr', m.mean_hr,
        'max_hr', m.max_hr,
        'min_hr', m.min_hr,
        'sdnn', m.sdnn
    ),
   
    -- 3. Собираем массив всех найденных аномалий (аритмий)
    'detected_anomalies', (
        SELECT json_agg(json_build_object(
            'anomaly_id', a.anomaly_id,
            'timestamp_start', a.timestamp_start,
            'timestamp_end', a.timestamp_end,
            'type', a.anomaly_type,
            'confidence', a.confidence_score
        ))
        FROM anomalies a
        WHERE a.record_id = r.record_id
    ),
   
    -- 4. Собираем массив отметок из дневника активности пациента
    'patient_log', (
        SELECT json_agg(json_build_object(
            'start_time', pl.start_time,
            'end_time', pl.end_time,
            'activity', pl.activity_type
        ))
        FROM patient_logs pl
        WHERE pl.record_id = r.record_id
    )
) AS screen_dataFROM records rJOIN patients p ON r.patient_id = p.patient_idLEFT JOIN ecg_metrics m ON r.record_id = m.record_idWHERE r.record_id = 98745; -- Сюда фронтендер будет подставлять нужный ID сессии

------------------------------


## Как устроен этот запрос внутри 

   1. json_build_object — специальная функция PostgreSQL. Она берет обычные строки из таблиц и на лету превращает их в пары "ключ": значение для JSON.
   2. json_agg — функция агрегации. Поскольку аномалий и записей в дневнике за сутки может быть очень много, эта функция собирает их все вместе и упаковывает в один аккуратный JSON-массив [...].
   3. LEFT JOIN — используется для таблицы метрик. Если Data Scientist еще не успел обработать запись и метрик в базе пока нет, запрос все равно сработает и отдаст врачу базовую информацию о пациенте, не ломая интерфейс.
   4. Фильтрация WHERE r.record_id = 98745 — делает запрос точечным. База данных не будет сканировать миллионы чужих записей, а мгновенно достанет данные только по текущему исследованию.

## В чем ценность для питониста-фронтендера:
Фронтендеру на Python достаточно будет выполнить этот один запрос из своего кода (через стандартные библиотеки вроде psycopg2 или asyncpg) и вызвать .json(). Он сразу получит ту самую структуру, которая была прописана в ТЗ, и сможет за один проход раскидать данные по виджетам и графикам Streamlit.

Теперь техническая цепочка полностью замкнулась: от файлов C++ и моделей Data Science данные легли в базу, а спроектированный  SQL-запрос идеально выводит их на экран врача.


Чтобы этот SQL-запрос работал за миллисекунды (как говорят в IT, «летал») даже на базах данных с миллионами строк, ИАС нужно спроектировать индексы для СУБД.
Без индексов PostgreSQL при каждом открытии экрана врача будет вынужден перебирать всю многомиллионную таблицу аномалий с начала до конца (это называется Sequential Scan). Индексы работают как алфавитный указатель в конце толстой книги — база данных сразу находит нужную страницу.
Вот конкретные команды для создания индексов, которые ИАС может добавить в проект и описать в как «Оптимизация производительности и индексирование СУБД»

------------------------------
## 1. Индексы для связывания таблиц (Внешние ключи)
Когда в запросе используются операторы JOIN или подзапросы WHERE ... = r.record_id, СУБД должна мгновенно связывать строки. Для этого нужно проиндексировать все внешние ключи (Foreign Keys).

-- Индекс для связи записей с пациентами (ускоряет JOIN patients)CREATE INDEX idx_records_patient_id ON records(patient_id);
-- Индекс для связи метрик с конкретной записью ХолтераCREATE INDEX idx_ecg_metrics_record_id ON ecg_metrics(record_id);

## 2. Покрывающие индексы для подзапросов (Самое важное для скорости)
В запросе есть два тяжелых подзапроса: поиск всех аномалий и всех записей дневника для конкретного record_id. Так как аномалий за сутки могут быть тысячи, обычного индекса мало. Нужно использовать покрывающий индекс (Covering Index) с ключевым словом INCLUDE.

-- Покрывающий индекс для таблицы аномалийCREATE INDEX idx_anomalies_record_id_incl ON anomalies(record_id) INCLUDE (anomaly_id, timestamp_start, timestamp_end, anomaly_type, confidence_score);
-- Покрывающий индекс для таблицы дневника пациентаCREATE INDEX idx_patient_logs_record_id_incl ON patient_logs(record_id) INCLUDE (start_time, end_time, activity_type);

## Почему это заставит базу «летать»? 
Благодаря ключевому слову INCLUDE, PostgreSQL применит технологию Index-Only Scan. Это значит, что база данных найдет все нужные данные (время аномалии, тип, уверенность) прямо внутри самого индекса и ей вообще не придется обращаться к жесткому диску к основной таблице anomalies. Скорость выполнения вырастет в десятки раз.


------------------------------
## 3. Бонус: Что делать с первичными ключами (Primary Keys)
ИАС важно помнить и указать , что для полей с типом PRIMARY KEY (таких как r.record_id в финальном условии WHERE) PostgreSQL автоматически создает уникальные индексы при создании таблиц. Дополнительно писать команду CREATE INDEX для них не нужно.

## Итог для команды
После того как ИАС применит этот скрипт:

   1. Запрос экрана врача будет занимать менее 5–10 миллисекунд даже на слабом сервере.
   2. Фронтендер сможет мгновенно обновлять дашборд при переключении между пациентами.
   3. Профессора на защите диплома увидят, что система спроектирована по всем канонам Enterprise-разработки Big Data.


Команда EXPLAIN ANALYZE в PostgreSQL — это главный инструмент инженера ИАС для проверки и доказательства того, что его индексы работают, а база данных не тормозит.
Если просто запустить SQL-запрос, база вернет данные. Но если перед запросом написать EXPLAIN ANALYZE, база данных выполнит запрос, замерит время каждой операции в миллисекундах и покажет детальный план его выполнения (Query Plan).
Для ИАС скриншоты и разбор логов EXPLAIN ANALYZE до и после создания индексов — это «железобетонное» доказательство его инженерной квалификации


------------------------------
## Шаг 1. Как запустить проверку
ИАС нужно взять спроектированный ранее SQL-запрос для экрана врача и просто дописать в самом начале две команды:

EXPLAIN (ANALYZE, BUFFERS)SELECT json_build_object(
    'record_id', r.record_id,
    -- ... (весь остальной текст нашего JSON-запроса) ...
)FROM records r-- ...WHERE r.record_id = 98745;

(Параметр BUFFERS очень важен: он показывает, сколько данных база прочитала из быстрой оперативной памяти, а сколько — с медленного жесткого диска)


------------------------------
## Шаг 2. Как понять, что индексы НЕ работают (До оптимизации)
Если индексы еще не созданы, в выданном логе ИАС  увидит страшные для любого инженера слова: Seq Scan (Sequential Scan — последовательное сканирование всей таблицы) и Filter.
Лог будет выглядеть примерно так:

->  Seq Scan on anomalies a  (cost=0.00..45230.10 rows=152 width=42)
    Filter: (record_id = 98745)
    Rows Removed by Filter: 2450120
Planning Time: 0.284 ms
Execution Time: 342.150 ms  <-- ОЧЕНЬ МЕДЛЕННО для одной операции!

Что это значит: Чтобы найти аномалии для одной сессии Холтера, база данных «ногами» прошлась по всем 2,5 миллионам строк в таблице anomalies, выкинула лишние (Rows Removed by Filter) и оставила нужные. Это заняло критические 342 миллисекунды. Если таких запросов будет много, сервер «ляжет».


------------------------------
## Шаг 3. Как понять, что индексы РАБОТАЮТ ИДЕАЛЬНО (После оптимизации)
После того как ИАС выполнит команды CREATE INDEX, он снова запускает EXPLAIN ANALYZE. Лог кардинально изменится. Теперь вместо Seq Scan он увидит заветные слова: Index Scan или Index Only Scan.
Лог превратится в такой:

->  Index Only Scan using idx_anomalies_record_id_incl on anomalies a  (cost=0.43..8.45 rows=152 width=42) (actual time=0.012..0.085 rows=152 loops=1)
    Index Cond: (record_id = 98745)
    Buffers: shared hit=4  <-- Прочитано всего 4 блока из кэша памяти!
Planning Time: 0.152 ms
Execution Time: 0.410 ms  <-- Ускорение почти в 1000 раз!

Что это значит для  стартапа:

   1. Index Only Scan — победа! База данных даже не заглядывала в саму таблицу на жестком диске. Она взяла все данные прямо из покрывающего индекса, который спроектировал ИАС.
   2. Buffers: shared hit=4 — база прочитала всего 4 крошечных блока данных из оперативной памяти. Нагрузка на процессор и диск равна нулю.
