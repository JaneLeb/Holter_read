## Что такое «дрейф изолинии»? 

<img width="320" height="240" alt="image" src="https://github.com/user-attachments/assets/7240de7e-427d-4b44-a691-9ea6eafd1d5d" />

Представьте, что вы рисуете кардиограмму фломастером на бумажной ленте.

* Изолиния --- это ровная горизонтальная линия, когда сердце отдыхает между ударами.
* Удар сердца --- это острый пик вверх и вниз, после которого график снова возвращается на эту ровную линию.

А теперь представьте, что вы рисуете ЭКГ в поезде, который трясется, или пациент с Холтером просто глубоко вздохнул, повернулся на бок или почесал грудь. В этот момент проводки смещаются, и весь ваш график вместе с ровной линией начинает плавно «плыть» вверх или падать вниз, как американские горки.
Вот это сползание всего графика целиком вверх или вниз и называется дрейфом изолинии.


Почему это плохо?
Робот (ИИ) или старая программа смотрят на этот уплывший вверх график и думают: «Ого, как высоко поднялся пик! Наверное, у пациента инфаркт!». Хотя на самом деле человек просто вздохнул, а само сердце бьется нормально


------------------------------
## Как убрать этот шум на C++?
Чтобы очистить график от этих «американских горок», используют специальные фильтры. Для новичка проще всего понять фильтр Скользящего Среднего (Moving Average).
Представьте, что мы берем текущую точку графика и смотрим на её «соседей» (например, 50 точек до неё и 50 точек после). Если мы посчитаем их среднее арифметическое, мы получим плавную линию той самой «горы», на которую уплыл наш график.
А теперь магия: если мы из нашего грязного (уплывшего) сигнала вычтем эту плавную линию горы, то график мгновенно выпрямится и снова станет ровным!

## Пример простейшего фильтра на C++\
Ниже написан максимально простой код. Он берет «гуляющий» график и выравнивает его.

      #include <iostream>#include <vector>\
      // Простой фильтр для выравнивания «гуляющего» графикаstd::vector<double> removeDrift(const std::vector<double>& dirtySignal, int windowSize) {\
          std::vector<double> cleanSignal = dirtySignal; // Создаем копию для чистого сигнала
      
          // Бежим по всем точкам графика (пропуская края, чтобы не вылететь за границы)\
          for (size_t i = windowSize; i < dirtySignal.size() - windowSize; ++i) {
      
              // 1. Считаем среднее арифметическое вокруг текущей точки (ищем "гору")\
              double sum = 0;\
              for (int j = -windowSize; j <= windowSize; ++j) {\
                  sum += dirtySignal[i + j];\
              }\
              double movingAverage = sum / (2 * windowSize + 1); // Вот она, плавная линия шума
      
              // 2. Вычитаем "гору" из грязной точки. График падает обратно на ровную линию!\
              cleanSignal[i] = dirtySignal[i] - movingAverage;\
          }
      
          return cleanSignal;\
      }\
      int main() {\
          // Симулируем грязный сигнал: сердце бьется около нуля (значения 0 и 10),\
          // но потом график плавно улетает вверх на гору (значения 100, 105) из-за вдоха.\
          std::vector<double> dirtySignal = {0, 10, 0, 5, 0, 100, 110, 100, 105, 100, 0, 5, 0};
      
          // Запускаем наш фильтр (окно сглаживания = 2 точки вправо и влево)\
          std::vector<double> cleanSignal = removeDrift(dirtySignal, 2);
      
          // Смотрим результат\
          std::cout << "Было (с дрейфом) -> Стало (чисто):\n";\
          for (size_t i = 2; i < cleanSignal.size() - 2; ++i) {\
              std::cout << dirtySignal[i] << " -> " << cleanSignal[i] << "\n";\
          }
      
          return 0;\
      }

Технический вопрос, на котором спотыкаются 90% начинающих разработчиков медицинского софта - что если пик дрейфа совпадет с R-пиком сердца?

Если «окно» нашего фильтра (количество точек для подсчета среднего) будет слишком маленьким, то фильтр примет высокий, острый R-пик за «гору» шума. Он попытается его сгладить и просто срежет (уничтожит) сам удар сердца. На выходе мы получим испорченную кардиограмму.
Чтобы этого не произошло, инженеры используют три хитрых трюка:


------------------------------
## Трюк 1. Размер имеет значение (Правильный подбор «окна»)
Вспомните, как выглядит ЭКГ:

* R-пик --- это очень быстрое, мгновенное событие. Сердце сжимается за доли секунды (около 0,1 секунды).
* Дрейф изолинии (вдох или поворот тела) --- это очень медленное событие. Человек вдыхает и выдыхает в течение 3--4 секунд.

Математическая магия в том, что мы делаем размер скользящего окна очень большим --- например, шириной в 1--2 секунды (это сотни точек сигнала).

* Когда такое огромное окно ползет по графику, короткий и узкий R-пик внутри него превращается в каплю в море. Он почти не влияет на среднее арифметическое значение всего окна.
* А вот медленная, огромная «гора» от дыхания отлично вычисляется. В итоге фильтр срезает только медленное дыхание, а быстрый R-пик оставляет целым и невредимым.

------------------------------
## Трюк 2. Медианный фильтр (Магия сортировки)
Вместо того чтобы считать среднее арифметическое (которое всё-таки немножко искажается из-за R-пиков), продвинутые программисты используют Медианный фильтр. Для вашей команды студентов, написавших тип Decimal, это будет супер-простая задача.
Как он работает:

   1. Мы берем точки внутри нашего окна.
   2. Сортируем их по возрастанию (от меньшего к большему).
   3. Берем точку, которая оказалась ровно посередине отсортированного списка (это и есть медиана).

Почему это идеально для ЭКГ?
Острый R-пик --- это «выброс» (одиночные очень большие значения). При сортировке эти огромные значения улетят в самый конец списка, а наш фильтр возьмет спокойную точку из середины. Таким образом, медианный фильтр физически игнорирует R-пики, видя только плавную линию дрейфа


------------------------------
## Трюк 3. Сначала найди, потом фильтруй (Адаптивный подход)
Самый профессиональный подход, который вы можете внедрить в свой стартап, чтобы утереть нос «Миокарду»:

   1. Алгоритм на C++ сначала делает черновой проход по сигналу и находит координаты всех R-пиков (это сделать легко, они самые высокие).
   2. На время работы фильтра очистки шума программа мысленно «вырезает» R-пики (заменяет их ровной линией).
   3. Фильтр спокойно очищает дрейф изолинии, не боясь ничего повредить.
   4. Программа возвращает R-пики на их законные места, но уже на идеально ровном графике.

в программировании медианный фильтр хорош тем, что в его основе лежит простая житейская логика: собрать соседей, построить их по росту и взять того, кто встал ровно посередине.
Чтобы этот фильтр работал очень быстро и эффективно на записях Холтера, в C++ лучше использовать стандартную функцию std::nth_element вместо полной сортировки всего массива. Она не сортирует весь список до конца, а только находит один нужный элемент для центра, что экономит кучу процессорного времени.
Ниже представлен простой и понятный код медианного фильтра.


## Код медианного фильтра на C++

#include <iostream>#include <vector>#include <algorithm> // Здесь живут функции для работы с массивами
// Функция медианного фильтраstd::vector<double> medianFilter(const std::vector<double>& dirtySignal, int windowSize) {
    std::vector<double> driftLine = dirtySignal; // Сюда запишем выделенную "гору" шума
    std::vector<double> cleanSignal = dirtySignal; // А сюда — итоговый чистый график

    // Бежим по графику, отступая от краев на размер окна
    for (size_t i = windowSize; i < dirtySignal.size() - windowSize; ++i) {
       
        // 1. Собираем всех соседей вокруг текущей точки в отдельный маленький вектор
        std::vector<double> windowPoints;
        windowPoints.reserve(2 * windowSize + 1);
       
        for (int j = -windowSize; j <= windowSize; ++j) {
            windowPoints.push_back(dirtySignal[i + j]);
        }

        // 2. Находим центральный элемент (медиану).
        // std::nth_element частично переставит элементы так, что в центре окажется правильное число.
        auto medianIterator = windowPoints.begin() + windowSize;
        std::nth_element(windowPoints.begin(), medianIterator, windowPoints.end());
       
        double medianValue = *medianIterator; // Вот она — чистая линия дыхания без R-пиков!
        driftLine[i] = medianValue;
    }

    // 3. Вычитаем полученную линию шума из грязного сигнала
    for (size_t i = windowSize; i < dirtySignal.size() - windowSize; ++i) {
        cleanSignal[i] = dirtySignal[i] - driftLine[i];
    }

    return cleanSignal;
}
int main() {
    // Симулируем ЭКГ-сигнал: спокойные точки (0),
    // затем резкий и огромный R-пик (500) — это удар сердца,
    // а затем график плавно уплывает вверх из-за дыхания (10, 12, 15, 14)
    std::vector<double> dirtySignal = {0, 0, 500, 0, 0, 10, 12, 500, 15, 14, 0, 0};

    // Запускаем фильтр. windowSize = 2 означает, что окно будет шириной 5 точек (2 слева + 1 центр + 2 справа)
    std::vector<double> cleanSignal = medianFilter(dirtySignal, 2);

    // Выводим результат в консоль
    std::cout << "Грязный сигнал -> Очищенный от дрейфа:\n";
    for (size_t i = 2; i < cleanSignal.size() - 2; ++i) {
        std::cout << dirtySignal[i] << " \t-> \t" << cleanSignal[i] << "\n";
    }

    return 0;
}

## Как этот код работает с R-пиком (разбор «на пальцах»)
Давайте посмотрим, что происходит, когда окно фильтра оказывается прямо на ударе сердца — на точке со значением 500:

   1. Программа собирает соседей вокруг этого пика. Допустим, получился набор: [0, 0, 500, 0, 10].
   2. Функция std::nth_element мысленно расставляет их по росту: [0, 0, 0, 10, 500].
   3. Ровно по центру (индекс 2) оказывается значение 0. Огромный пик 500 отброшен в самый конец и проигнорирован!
   4. В качестве значения «шума» для этой точки фильтр выбирает 0.
   5. Магия вычитания: 500 (оригинал) - 0 (шум) = 500. Наш R-пик остался абсолютно целым, а вот медленное сползание графика к значениям вроде 12 и 15 превратится обратно в четкие нули.

В реальном Холтере миллионы точек, и создавать новый вектор windowPoints на каждом шаге цикла — это медленно. Поэтому для реального проекта нужно переписать алгоритм так, чтобы использовать один и тот же буфер памяти или применят алгоритм движущейся медианы (Running Median), который работает вообще мгновенно.

Тестирование дрейфа
Для написания автотестов в C++ профессиональные разработчики обычно используют готовые библиотеки вроде Google Test (gtest) или Catch2. Вот простой, понятный и легковесный тестовый фреймворк прямо в коде.
Это не потребует сложной настройки проекта и позволит сразу проверить, не ломается ли логика фильтра при изменениях в коде.

## Как устроен автотест (Простыми словами)
Идея автотеста состоит из трех шагов:

   1. Задать входные данные (Arrange): Мы сами вручную придумываем «грязный» массив точек, для которого точно знаем, каким должен быть идеальный результат.
   2. Запустить фильтр (Act): Передаем эти данные в нашу функцию medianFilterFast.
   3. Сверить результат (Assert): Проверяем, совпали ли полученные числа с нашими ожиданиями. Если совпали --- тест пройден, если нет --- программа бьет тревогу.

Ниже представлен готовый код с тремя важнейшими тестами для медицинского софта.
## Код автотестов на C++

      #include <iostream>#include <vector>#include <cmath> // Здесь живет функция std::abs для сравнения дробных чисел#include <algorithm>
      
      
            // Вставляем сюда нашу оптимизированную функцию фильтра из прошлого шагаstd::vector<double> medianFilterFast(const std::vector<double>& dirtySignal, int windowSize) {\
                size_t signalSize = dirtySignal.size();\
                std::vector<double> cleanSignal(signalSize);\
                size_t windowLength = 2 * windowSize + 1;\
                std::vector<double> windowBuffer(windowLength);\
                auto medianIterator = windowBuffer.begin() + windowSize;
            
                for (size_t i = windowSize; i < signalSize - windowSize; ++i) {\
                    std::copy(dirtySignal.begin() + (i - windowSize),\
                              dirtySignal.begin() + (i + windowSize + 1),\
                              windowBuffer.begin());\
                    std::nth_element(windowBuffer.begin(), medianIterator, windowBuffer.end());\
                    cleanSignal[i] = dirtySignal[i] - *medianIterator;\
                }\
                return cleanSignal;\
            }\
            // Вспомогательная функция для сравнения двух векторов с плавающей точкойbool areVectorsEqual(const std::vector<double>& v1, const std::vector<double>& v2, double epsilon = 0.0001) {\
                if (v1.size() != v2.size()) return false;\
                for (size_t i = 2; i < v1.size() - 2; ++i) { // Проверяем только рабочую зону (без краев)\
                    if (std::abs(v1[i] - v2[i]) > epsilon) {\
                        return false;\
                    }\
                }\
                return true;\
            }\

                    std::cout << "[ОК] Тест 1: Идеальный сигнал не портится.\n";\

                    std::cout << "[ОШИБКА] Тест 1: Фильтр изменил чистый сигнал!\n";\

            }\

                // График уплыл вверх ровно на 50 единиц (0 + 50, 10 + 50...)\
                std::vector<double> driftedSignal = {50, 60, 50, 55, 50, 60, 50};\
                // Ожидаем, что после вычитания "горы" в 50 единиц график вернется к норме\
                std::vector<double> expected      = { 0, 10,  0,  5,  0, 10,  0};
            
                std::vector<double> result = medianFilterFast(driftedSignal, 2);

                    std::cout << "[ОК] Тест 2: Постоянный дрейф изолинии успешно удален.\n";\

                    std::cout << "[ОШИБКА] Тест 2: График не выровнялся!\n";\

## Почему это важно для стартапа?
Когда ваша команда начнет дорабатывать и ускорять код (например, переписывать его под многопоточность или оптимизировать алгоритм Хонга), они могут случайно перепутать индексы и сломать логику.
Если у вас будут написаны такие автотесты, студентам достаточно будет нажать одну кнопку запуска:

* Если все загорелось зеленым ([ОК]) --- значит, их оптимизация безопасна.
* Если тест упал --- они сразу увидят ошибку до того, как этот код попадет к вашей сестре-кардиологу на реальные файлы пациентов. В медицине такой подход (TDD --- разработка через тестирование) критически важен.


Чтобы ускорить этот код и сделать его пригодным для обработки гигантских файлов Холтера (где содержатся миллионы точек), нужно убрать из цикла операцию std::vector<double> windowPoints. В C++ выделение памяти в куче (heap allocation) внутри тяжелого цикла --- это главная причина тормозов.
Мы можем сделать умный ход: выделить память под «окошко» всего один раз до начала цикла, а внутри цикла просто перезаписывать данные в этом готовом буфере.
Ниже представлен оптимизированный вариант, который будет работать в разы быстрее.
## Оптимизированный код на C++

               #include <iostream>#include <vector>#include <algorithm>\
               // Быстрый медианный фильтр без выделения памяти в циклеstd::vector<double> medianFilterFast(const std::vector<double>& dirtySignal, int windowSize) {\
                   size_t signalSize = dirtySignal.size();\
                   std::vector<double> cleanSignal(signalSize); // Выделяем память под результат один раз
               
                   // 1. Выделяем буфер под окно ВСЕГО ОДИН РАЗ до цикла.\
                   // Размер окна: windowSize слева + 1 центр + windowSize справа\
                   size_t windowLength = 2 * windowSize + 1;\
                   std::vector<double> windowBuffer(windowLength);
               
                   // Итератор, указывающий на центральный элемент в буфере (медиану)\
                   auto medianIterator = windowBuffer.begin() + windowSize;
               
                   // 2. Бежим по сигналу\
                   for (size_t i = windowSize; i < signalSize - windowSize; ++i) {
               
                       // Вместо создания нового вектора, просто копируем данные в уже существующий буфер.\
                       // Используем быстрый std::copy для копирования куска памяти.\
                       std::copy(dirtySignal.begin() + (i - windowSize),\
                                 dirtySignal.begin() + (i + windowSize + 1),\
                                 windowBuffer.begin());
               
                       // Находим медиану. std::nth_element работает прямо внутри нашего постоянного буфера.\
                       std::nth_element(windowBuffer.begin(), medianIterator, windowBuffer.end());
               
                       // Вычитаем найденную линию шума из грязного сигнала прямо на лету\
                       cleanSignal[i] = dirtySignal[i] - *medianIterator;\
                   }
               
                   return cleanSignal;\
               }\
               int main() {\
                   // Тестовый сигнал\
                   std::vector<double> dirtySignal = {0, 0, 500, 0, 0, 10, 12, 500, 15, 14, 0, 0};
               
                   // Запускаем быстрый фильтр\
                   std::vector<double> cleanSignal = medianFilterFast(dirtySignal, 2);
               
                   std::cout << "Очищенный сигнал (быстрый алгоритм):\n";\
                   for (size_t i = 2; i < cleanSignal.size() - 2; ++i) {\
                       std::cout << cleanSignal[i] << "\n";\
                   }
               
                   return 0;\
               }

## За счет чего код стал быстрее?

   1. Реаллокация памяти равна нулю: В прошлом варианте на каждой итерации цикла операционная система тратила время на поиск свободного кусочка памяти под windowPoints, а в конце итерации удаляла его. Теперь windowBuffer выделен один раз в самом начале и просто повторно использует свои ячейки памяти.
   2. Процессорный кэш ликует (Data Locality): Поскольку windowBuffer постоянно лежит в одном и том же месте памяти, процессор перемещает его в свой сверхбыстрый кэш (L1/L2). Чтение и запись происходят мгновенно.
   3. Объединение циклов: В первой версии мы сначала создавали массив шума, а потом вторым циклом вычитали его. Здесь мы находим медиану и сразу же вычитаем её из оригинальной точки за один проход.

## Идея на «вырост» (Уровень: Мидл)
Если  выжать абсолютно максимальную скорость (чтобы обрабатывать суточный Холтер за миллисекунды), нужно применить концепцию Алгоритма движущейся медианы Хонга (Huang's algorithm) или использование структуры данных «Гистограмма».
Поскольку при движении окна на одну точку вправо у нас из окна уходит всего один старый элемент и добавляется всего один новый, нам не обязательно каждый раз заново запускать std::nth_element. Можно использовать специальную структуру (например, сбалансированное дерево поиска или гистограмму), которая умеет выкидывать старый элемент и вставлять новый за очень малое количество шагов.
